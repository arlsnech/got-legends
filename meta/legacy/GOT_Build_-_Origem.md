## 📦 Explicando cada JSON — o que é, pra que serve, e como otimizar

Você perguntou sobre `weapons.json` e se todos os arquivos são necessários. A resposta honesta: **não precisamos de 6 arquivos separados**. Podemos fazer com 3, que são mais inteligentes e diretos. Deixa eu explicar o que cada um proposto faria, e o que proponho no lugar:

---

### O que eu tinha proposto (e o problema)

```
classes.json      → dados das 4 classes
techniques.json   → técnicas de todas as classes
properties.json   → todas as propriedades de todos os equipamentos
perks.json        → todas as vantagens de todos os equipamentos
weapons.json      → info dos equipamentos (katanas, arcos, AF...)
legendary.json    → itens magistrais separados
```

**O problema:** `weapons.json` seria uma lista de equipamentos COM suas propriedades e vantagens embutidas. Isso significa que os dados de propriedades/vantagens de um mesmo equipamento estariam duplicados — uma vez em `weapons.json` (referenciando as opções disponíveis) e outra em `properties.json` e `perks.json` (com os detalhes). Redundância desnecessária.

E `legendary.json` separado é de fato supérfluo, como você intuiu. Um magistral **é** um equipamento — só tem flag especial.

---

### O que proponho: 3 JSONs

```
data.json
  ├── classes[]
  ├── techniques[]
  └── gear[]          ← tudo: katanas, alcance, amuletos, AF, magistrais
```

Ou, se quiser manter separado por performance e organização:

```
classes.json     → classes + técnicas juntas (são inseparáveis)
gear.json        → todos os equipamentos (normais + magistrais unificados)
stats.json       → definição das estatísticas (pra montar o painel)
```

---

### Detalhando cada arquivo

**`classes.json`** — Contém as 4 classes com:
```json
{
  "id": "samurai",
  "name": { "pt": "Samurai", "en": "Samurai" },
  "baseHP": 100,
  "baseResolve": 3,
  "ultimate": { "name": "Fúria de Hachiman", "cost": 3, "strikes": 7 },
  "abilities": ["Extração Espiritual", "Lâmina Explosiva", "Chama Furiosa"],
  "techniques": [ ...lista das técnicas com seus efeitos em statKeys... ]
}
```

As técnicas ficam **dentro** da classe porque são sempre filtradas por classe. Não faz sentido ter um arquivo separado onde você vai filtrar `where classe == "Samurai"` toda vez.

---

**`gear.json`** — Contém TODOS os equipamentos, incluindo magistrais como flag dentro do mesmo objeto. Exemplo:

```json
{
  "id": "katana_petrea",
  "name": { "pt": "Katana Pétrea", "en": "Stone Katana" },
  "type": "katana",
  "subtype": "stone",
  "legendary": false,
  "cooldown": null,
  "ammo": null,
  "properties": [
    {
      "id": "dano_cc",
      "name": { "pt": "Dano Corpo a Corpo", "en": "Melee Damage" },
      "slots": ["P1", "P2"],
      "min": 0.05,
      "max": 0.12,
      "unit": "%",
      "statKey": "meleeDamage"
    }
  ],
  "perks": [
    {
      "id": "lamina_ardente",
      "name": { "pt": "Lâmina Ardente", "en": "Burning Blade" },
      "description": { "pt": "...", "en": "..." },
      "statEffects": [],
      "infoOnly": true
    }
  ]
}
```

E o Kunai Espiritual seria:
```json
{
  "id": "kunai_espiritual",
  "name": { "pt": "Kunai Espiritual", "en": "Spirit Kunai" },
  "type": "ghostWeapon1",
  "basedOn": "kunai",
  "legendary": true,
  "tier": "S",
  "cooldown": 30,
  "exclusivePerk": {
    "description": { "pt": "Abater com Kunai reduz TODAS as recargas em 10s", "en": "..." }
  },
  "properties": "inherit:kunai",  // herda as props do Kunai base no max
  "fixedPropertyValues": "max"    // magistral sempre no máximo
}
```

Isso resolve sua dúvida: o magistral referencia o equipamento base via `basedOn`, herda as propriedades no máximo, e a ferramenta bloqueia a edição dos valores.

---

**`stats.json`** — Define quais estatísticas existem, como calcular, e como exibir. Esse arquivo é o "cérebro" do painel:

```json
{
  "id": "meleeDamage",
  "label": { "pt": "Dano Corpo a Corpo", "en": "Melee Damage" },
  "unit": "%",
  "base": 0,
  "category": "damage",
  "stackingRule": "additive",
  "displayAlways": true,
  "sources": ["property", "technique"]
}
```

## 📐 Regras de empilhamento (stacking) — o que acumula e o que não acumula

Do site de dano + planilha da comunidade:

| Fonte | Acumula? | Regra |
|---|---|---|
| Qualquer % de dano da mesma categoria | ✅ Sim | ADITIVO — somam linearmente |
| Fontes de fogo de jogadores diferentes | ✅ Sim | Cada jogador pode aplicar o seu |
| Duas propriedades iguais no mesmo item | ❌ Não | Jogo bloqueia P1 = P2 |
| Enfraquecimento de múltiplas fontes (efeito de -25% dano) | ❌ Não | Só 1 instância conta |
| Barra de Atordoante com Enfraquecimento | ✅ Sim | 2 dardos = 2x velocidade de redução |
| Raio de Explosão % (propriedade) + Flecha Explosiva | ✅ Sim | Afeta bomba aderente e flecha explosiva |
| Raio Aumentado (perk da Bomba Aderente) + outros | ❌ Não | Só ajuda a própria bomba |
| Fumaça da Bomba de Magma + Raio | ❌ Não | Fumaça não muda de tamanho |
| Regeneração Rápida (Ronin) + Regeneração em Combate (perk) | ✅ Sim | Aumenta a taxa de regeneração passiva |
| Arremessos Impactantes + Onda Atordoante do Ronin | ✅ Sim | Derrubados = vulneráveis a críticos |

## 🌐 Sobre o multilinguismo no Google Sheets

Sim, é tecnicamente possível com Google Apps Script criar um menu que troca o idioma dos dados visíveis. Mas para uma planilha de banco de dados como essa, seria um script que intercalaria duas versões dos textos usando arrays nomeados — complicado e frágil.

**O que eu recomendo:**
1. Manter a planilha PT como primária (que você já tem)
2. Gerar a versão EN a partir dela — eu faço isso, associando cada campo ao termo correto do site ghostoftsushimalegends.com e do ghostfranchise.fandom.com
3. Publicar as duas como Google Sheets separadas (uma PT, uma EN)
4. Na ferramenta React: o JSON já terá `{ "pt": "...", "en": "..." }` para cada campo, e um botão no topo alterna o idioma instantaneamente — **sem precisar de dois JSONs**

## Por que estava em branco no Netlify e Vercel

O `vite.config.js` tinha `base: '/got-legends/'`. Isso fazia o build gerar um `index.html` assim:

```html
<script src="/got-legends/assets/index-XXXX.js"></script>
```

No **GitHub Pages** isso funciona porque o site fica em `arlsnech.github.io/got-legends/` — o caminho bate.

No **Netlify/Vercel** o site fica na raiz `got-legends.netlify.app/`, então o browser tenta carregar `got-legends.netlify.app/got-legends/assets/...` — que não existe → tela em branco.

---

## Como corrigir agora (3 passos)

### Passo 1 — Copie os 3 arquivos para a pasta do projeto

Baixe os 3 arquivos acima e coloque em `C:\Users\alexk\got-legends\`:
- `vite.config.js` → substitui o que já existe
- `netlify.toml` → arquivo novo na raiz
- `vercel.json` → arquivo novo na raiz

### Passo 2 — Suba para o GitHub

```powershell
cd C:\Users\alexk\got-legends
git add vite.config.js netlify.toml vercel.json
git commit -m "fix: base path for Netlify and Vercel"
git push
```

O Netlify e o Vercel vão detectar o push automaticamente, fazer um novo deploy (em ~1 minuto cada) e funcionar.

### Passo 3 — Deploys futuros para GitHub Pages

Toda vez que quiser atualizar o GitHub Pages, use este comando que define o base correto antes de buildar:

```powershell
$env:VITE_BASE_URL="/got-legends/"; npm run build; cd dist; git init; git add .; git commit -m "deploy"; git push --force https://github.com/arlsnech/got-legends.git HEAD:gh-pages; cd ..
```

# 📊 Relatório Completo de Plataformas de Hospedagem

## Estado atual dos seus sites

✅ GitHub Pages → `arlsnech.github.io/got-legends/`

## 1. Análise das plataformas — comparação técnica atualizada (2026)

### Plataformas que você já usa

**Cloudflare Pages** tem o free tier mais generoso de todos: bandwidth ilimitado, sem asteriscos. Vercel oferece 100 GB, GitHub Pages 100 GB, Netlify ~30 GB (baseado em créditos desde setembro de 2025).

**Netlify** moveu para um sistema de créditos em setembro de 2025. Cada deploy, GB de bandwidth e GB-hora de computação consome créditos a taxas diferentes, tornando a previsão de custo mais difícil. O overage de bandwidth é notavelmente caro a $55/100 GB, versus ~$15/100 GB no Vercel.

Notável: Netlify reduziu os build minutes gratuitos de 300 para 100 em 2025, tornando-o menos atraente para projetos pessoais.

### Plataformas solicitadas para análise

| Plataforma | Tipo | Free tier | Verdict |
|---|---|---|---|
| **Cloudflare Pages** | Static/Edge | ✅ Ilimitado (bandwidth + sites + builds 500/mês) | **Recomendada fortemente** |
| **Firebase Hosting** | Static | ⚠️ 10 GB storage, 360 MB/dia egress | Fraco para crescimento |
| **Render** | PaaS fullstack | ✅ Web services + PostgreSQL free | Excelente para backend |
| **Railway** | PaaS | ⚠️ $5 trial (não é mais "always free") | Bom para testar, pago depois |

**Cloudflare Pages** é a resposta quando seu tráfego é esporádico, viral, ou simplesmente grande. Bandwidth gratuito ilimitado não é marketing — é o plano real. Aceite um workflow um pouco mais CLI-driven e você tem uma economia que ninguém nessa lista consegue igualar.

**Firebase Hosting** só faz sentido se você já está comprometido com Firebase. Se Firestore e Firebase Auth estão alimentando seu app, hospedar junto é sem fricção. Se não estiverem, o limite de 10 GB de bandwidth e deploys apenas via CLI tornam esta a opção mais fraca de hospedagem pura.

**Render** oferece web services gratuitos com 512 MB RAM, PostgreSQL gratuito e domínios customizados — a escolha certa para apps fullstack.

## 2. Mapa de escolha por tipo de projeto

| Projeto | Recomendação |
|---|---|
| Site estático / SPA React | **Cloudflare Pages** (1°) → GitHub Pages (backup) |
| App Next.js com SSR | Vercel |
| Backend Node.js + banco de dados | Render (free) → Railway (pago) |
| App fullstack complexo | Cloudflare Pages (front) + Supabase (banco) |
| Wiki/guia com muitas imagens | Cloudflare Pages + Cloudflare R2 (storage) |

## 3. Visão para projetos futuros mais complexos

Para as ferramentas que você mencionou (bestiário, wiki, crafting, mapa interativo, to-do de missões com banco de dados), a stack ideal gratuita é:

```
Frontend (React/Vite)    → Cloudflare Pages
Banco de dados           → Supabase (PostgreSQL gratuito, 500 MB)
Imagens/Assets           → Cloudflare R2 ou Supabase Storage
Auth (login de usuário)  → Supabase Auth (gratuito)
API/funções backend      → Cloudflare Workers (100k req/dia gratuito)
```

A stack zero-custo recomendada para static + backend: GitHub Pages + Cloudflare Workers + Turso (banco SQLite edge).

## 4. Publicar no Cloudflare Pages — passo a passo completo

### 4.1 Criar conta na Cloudflare (gratuito)

Acesse **https://dash.cloudflare.com/sign-up** e crie uma conta. Não precisa de cartão de crédito.

### 4.2 Preparar o arquivo `_redirects`

Baixe o arquivo `_redirects` dos outputs acima. Copie-o para a pasta `public/` do seu projeto:

```
C:\Users\alexk\got-legends\public\_redirects
```

(Se a pasta `public/` não existir, crie-a.) O Vite copia automaticamente tudo de `public/` para `dist/` no build — então o `_redirects` vai estar no lugar certo para o Cloudflare Pages interpretar.

### 4.3 Subir o arquivo para o GitHub

```powershell
cd C:\Users\alexk\got-legends
git add public/_redirects
git commit -m "fix: add _redirects for Cloudflare Pages SPA routing"
git push
```

### 4.4 Conectar no Cloudflare Pages

1. Acesse **https://dash.cloudflare.com/** → vá em **Workers & Pages** no menu lateral
2. Clique **Create** → **Pages** → **Connect to Git**
3. Conecte sua conta GitHub (autorize a Cloudflare a acessar seus repositórios)
4. Selecione o repositório **`arlsnech/got-legends`**
5. Clique **Begin setup**

### 4.5 Configurar o build

Na tela "Set up builds and deployments", preencha:

| Campo | Valor |
|---|---|
| **Project name** | `got-legends` |
| **Production branch** | `main` |
| **Framework preset** | `Vite` (se aparecer) |
| **Build command** | `npm run build` |
| **Build output directory** | `dist` |

Clique **Save and Deploy**.

### 4.6 URL do Cloudflare Pages

Após o deploy, você receberá um subdomínio único no formato `*.pages.dev`. Neste caso será **`got-legends.pages.dev`** (ou um nome gerado automaticamente se já existir).

A partir daí, cada `git push` para o `main` dispara rebuild e deploy automático. O Cloudflare também gera preview URLs para cada Pull Request.
