# ⚠️ Dúvidas, Inconsistências e Pontos para Alinhar

### 1. Cor de classe no texto copiado — impossível em plain text
Texto copiado para a área de transferência é sempre **texto puro** (plain text). Nenhuma plataforma de chat (Discord, Reddit, WhatsApp, etc.) suporta cor em texto colado. O que **é** possível:
- `**NomeClasse**` → **negrito** no Discord e Reddit (Markdown)
- Emoji de classe (⚔️ Samurai, 🏹 Caçadora, 🐕 Ronin, 🗡️ Assassino) como substituto visual de cor

No **print/imagem** gerado, cor por classe fica perfeito. Sugiro: texto = negrito + emoji, imagem = cor completa de classe.

### 2. Nome da build para exportação — estado local x compartilhado
O `buildName` é atualmente estado **local** do `SavePanel`. Para os botões de exportação acessarem esse nome, preciso **elevar** (`lift`) esse estado para o `App`. Isso é uma mudança simples (3 linhas), mas preciso fazê-la antes de tudo. Confirma que isso pode ser feito? (Não muda nenhum comportamento visível — o campo de nome continua no mesmo lugar.)

### 3. Ordem dos botões — confirmando o que entendi
Você disse que quer: **Build → Detalhado → Estatístico** (Build primeiro). Confirmo essa ordem.

### 4. Estrutura dos 6 sub-botões (texto + imagem por modo)
Entendi que cada modo teria **dois botões lado a lado**:
```
[📋 Build]    [🖼️]   [📋 Detalhado]    [🖼️]   [📋 Estatístico]    [🖼️]
```
Ou seja, 3 pares de botões. O `📋` copia o texto, o `🖼️` gera a imagem. Confirmo esse entendimento antes de implementar.

### 5. Imagem: Canvas API (sem dependência) vs html2canvas
**Canvas API pura** (recomendado):
- Nenhuma dependência nova, funciona em qualquer browser
- Exige programar manualmente o layout desenhado no canvas
- Resultado: imagem PNG customizável, com cores de classe, boxes, etc.
- **Limitação**: texto muito longo pode precisar de quebra de linha manual (implementável)

**html2canvas** (alternativa):
- `npm install html2canvas` (pacote de ~300KB)
- Renderiza um elemento HTML do DOM como imagem (pixel-perfect com a UI)
- Mais fácil de implementar a primeira versão
- Limitação: não suporta bem elementos fora da viewport; pode ter bugs com `position: fixed`

**Minha recomendação**: Canvas API para máximo controle e zero dependência. Faço um layout de imagem personalizado (3 colunas, cores de classe, caixinhas, tipografia) — fica muito mais bonito do que uma captura da própria UI.

### 6. Responsive no layout 3 colunas
O breakpoint atual é 900px → 1 coluna. Com 3 colunas, sugiro:
- ≥ 1100px: 3 colunas (260px | 1fr | 340px)
- 700–1099px: 2 colunas (1fr | 320px) — técnicas+equipamentos juntos, stats separado
- < 700px: 1 coluna empilhado

### 7. Recarga da habilidade no texto — já calculada
Para mostrar "Habilidade de Classe: nome (descrição) [Xs recarga]" com a recarga **já reduzida por CDR**, o `stats.abilityCooldown.finalCd` já existe e é calculado em `logic.js`. Perfeito — já está tudo lá.

### 8. Estatísticas "somente modificadas" — como identificar
No `getStatGroups`, cada stat tem `{ value, base }`. `value !== base` = stat modificado. Na lógica atual, `base` para stats numéricos é `0`. Então `value !== 0` = modificado. Mas **HP e Determinação** sempre têm valor base (100 e 3 respectivamente). Vou tratar isso: só mostrar HP se > 100, Determinação se > 3.

# 🏗️ FASE 1 — Novo Layout 3 Colunas

## Como funciona a mudança

O novo layout usa **Flexbox vertical** no container principal (ao invés do `minHeight: 100vh` simples atual). Assim:
- O `header` é `flex-shrink: 0` — ocupa apenas o que precisa
- O `top-bar` (SavePanel + botões de exportação) é `flex-shrink: 0`
- A área das 3 colunas é `flex: 1; overflow: hidden` — ocupa **todo o espaço restante**
- Cada coluna individual tem `overflow-y: auto` — scroll independente

Isso elimina o problema atual de "ter que rolar muito para configurar a build" porque cada seção tem seu próprio scroll.
